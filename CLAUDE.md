# CLAUDE.md

Project conventions for Claude Code / Claude agents working in this repo. See `PLAN.md` for the full phased plan and EKS→AKS mapping.

## What this is

A multi-tenant AKS cluster: Terraform provisions Azure, ArgoCD GitOps-bootstraps everything above the API server. Port of a sibling EKS project (see `README.md`) — same shared-responsibility model, same tenant onboarding contract, different cloud plumbing.

## Stack

AKS · Terraform (`azurerm`, `azapi` fallback for NAP) · ArgoCD · Node Auto Provisioning (Karpenter) · Istio ambient · Gateway API · cert-manager · External Secrets Operator · Kyverno · Cloudflare (DNS/TLS)

## Repo layout

```
infra/            Terraform: network, aks, identity, keyvault modules (S3 remote state, see backend.hcl.example)
argocd/bootstrap/ app-of-apps tree: platform/ then tenants/
tenants/          <team>.json onboarding files (source of truth for ApplicationSet)
demo-app/         team-a (stateless), team-b (stateful) example workloads
docs/             tenant guides, platform runbooks, dev/design docs
```

## Conventions

- **Naming**: Azure resources `mtc-aks-dev-<resource>` (e.g. `mtc-aks-dev-rg`, `mtc-aks-dev-vnet`).
- **Terraform boundary**: Terraform stops at the API server (cluster, network, identity, Key Vault). It does NOT install Helm releases for platform capabilities — ArgoCD owns everything above the API server. Don't add `helm_release` resources to `infra/` for Istio/cert-manager/ESO/Kyverno/etc.
- **Workload classes**: node/pod labels use `workload-class: general|database|gpu` — keep this label contract identical to the EKS reference so tenant manifests are portable.
- **Tenant contract**: onboarding a team is exactly one JSON file at `tenants/<team_name>.json` with `name`, `sourceRepo`, `manifestPath`. Don't hand-write namespaces/AppProjects/ApplicationSets — those are generated by the ArgoCD ApplicationSet git-file generator.
- **Sync-wave order** (ArgoCD): CRDs/Gateway API → Istio → cert-manager/ESO/Kyverno → external-dns → NodePools/StorageClasses → tenants. Respect existing `argocd.argoproj.io/sync-wave` annotations when adding platform manifests.
- **TF state backend**: S3, not Azure Storage — same shared bucket used across this project's sibling AWS/EKS project and other cloud projects, distinguished by `key` prefix (`multi-tenant-aks/<env>/terraform.tfstate`). Locking is S3-native (`use_lockfile = true`), no DynamoDB. Real bucket/region live in `infra/backend.hcl` (gitignored) — copy from `infra/backend.hcl.example`.
- **Secrets**: never commit secrets or static credentials. All secrets flow from Azure Key Vault via ESO + Workload Identity federation. No secrets in Terraform state variables beyond what Azure itself requires.
- **DNS/TLS**: Cloudflare is the DNS provider and DNS-01 solver; domain is `arguswatcher.net`. Don't introduce Azure DNS zones without updating `PLAN.md`.

## Common commands

```sh
# One-time per machine/checkout: point at the shared S3 state bucket
cp infra/backend.hcl.example infra/backend.hcl   # then fill in bucket/region — gitignored, never commit

# Terraform
terraform -chdir=infra init -backend-config=backend.hcl
terraform -chdir=infra fmt -recursive
terraform -chdir=infra validate
terraform -chdir=infra plan
terraform -chdir=infra apply

# Cluster access
az aks get-credentials --resource-group mtc-aks-dev-rg --name mtc-aks-dev
kubectl get nodes -L workload-class

# GitOps bootstrap
kubectl apply -f app-of-apps.yaml
kubectl -n argocd port-forward svc/argocd-server 8080:443
```

## Do / Don't

- Do keep tenant-facing docs (`docs/tenant/`) in sync whenever a capability's interface changes (nodeSelector keys, StorageClass names, URL pattern).
- Do run the relevant Phase acceptance check (see `PLAN.md` §4) before considering a phase done.
- Don't touch AKS managed Istio add-on — this project uses self-managed Istio ambient via Helm; the add-on doesn't support ambient mode.
- Don't add tenant-specific logic to platform manifests — tenant differences belong in `tenants/*.json` and the tenant's own repo.
